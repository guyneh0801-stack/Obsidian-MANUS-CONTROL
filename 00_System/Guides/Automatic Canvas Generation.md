# Automatic Canvas Generation

מנגנון זה מייצר ומעדכן קובצי Canvas אוטומטיים מתוך פתקי Relationship בתיקיית `04_Relationships`. המטרה היא לספק לך גרפי קשרים ממוקדים לכל תיק חקירה, בנוסף למפה גלובלית, ולאפשר לך לפתוח אותם ב-Obsidian, להזיז צמתים ידנית, ולהריץ שוב את העדכון בלי לאבד את הסידור שבנית.

## קובצי הפלט

הסקריפט ייצור מספר קבצי Canvas, רשימות קשרים ולוח בקרה מרכזי בתיקיית `04_Relationships`:

| קובץ | תיאור |
| --- | --- |
| `Relationship Map Dashboard.md` | לוח בקרה מרכזי לפתיחת המפה הגלובלית או מפת תיק ספציפי בלחיצה מתוך Obsidian. |
| `Auto Relationship Map.canvas` | מפת קשרים גלובלית המכילה את כל הקשרים מכל תיקי החקירה. במפה זו יופיעו גם קבוצות וצבעים לפי תיק. |
| `Relationship Edge List.md` | רשימת טקסט של כל הקשרים הגלובליים. |
| `Case - [שם_תיק] Relationship Map.canvas` | מפת קשרים ספציפית לכל תיק חקירה שצוין בפתקי הקשר. זו התצוגה הנקייה ביותר לעבודה ממוקדת. |
| `Case - [שם_תיק] Relationship Edge List.md` | רשימת טקסט של הקשרים בתיק חקירה ספציפי. |

דוגמאות:
```text
MANUS CONTROL/04_Relationships/Relationship Map Dashboard.md
MANUS CONTROL/04_Relationships/Auto Relationship Map.canvas
MANUS CONTROL/04_Relationships/Case - Project_Alpha Relationship Map.canvas
MANUS CONTROL/04_Relationships/Case - Project_Beta Relationship Map.canvas
```

## קובץ הסקריפט

```text
MANUS CONTROL/00_System/Scripts/generate_relationship_canvas.py
```

## איך זה עובד

הסקריפט קורא כל קובץ Markdown בתיקיית `04_Relationships`, מחפש frontmatter עם השדות הבאים, וממיר כל פתק קשר ל-edge ב-Canvas. פתקי הקשר הם מקור האמת עבור הקשתות, ואילו קובץ ה-Canvas הקיים הוא מקור האמת עבור מיקומי הצמתים הידניים:

| Field | Purpose |
| --- | --- |
| `from_entity` | node מקור |
| `to_entity` | node יעד |
| `relationship_type` | label על החץ |
| `confidence` | משפיע על צבע/סיווג |
| `status` | hypothesis / probable / confirmed / rejected |
| `linked_sources` | נשמר בפתק המקורי כראיית תמיכה |
| `related_case` | שם תיק החקירה אליו משויך הקשר. משמש ליצירת מפות ממוקדות. |

## איך לבחור מה לראות ב-Obsidian

Obsidian Canvas אינו כולל מנגנון סינון דינמי מובנה שמסתיר ומציג תיקים בתוך אותו קובץ Canvas בזמן אמת. לכן המערכת משתמשת בשיטה יציבה יותר: היא מייצרת **מפה נפרדת לכל תיק חקירה**, ובמקביל מייצרת **Relationship Map Dashboard** שמרכז את כל קישורי התצוגה במקום אחד.

| צורך אנליטי | הקובץ לפתיחה | ההיגיון המבצעי |
| --- | --- | --- |
| לראות רק תיק חקירה אחד | `Case - [שם_תיק] Relationship Map.canvas` | הקובץ מכיל רק את הקשרים שבהם `related_case` שווה לשם התיק. |
| לעבור מהר בין תיקים | `Relationship Map Dashboard.md` | לוח הבקרה מציג טבלה של כל התיקים, קישור ל-Canvas, קישור לרשימת הקשרים וספירת קשרים/ישויות. |
| לראות תמונת על בין כל התיקים | `Auto Relationship Map.canvas` | המפה הגלובלית כוללת את כל הקשרים, אבל מוסיפה קבוצות וצבעים לפי תיק כדי להפחית ערבוב ויזואלי. |
| לבדוק קשרים כטבלה | `Relationship Edge List.md` או `Case - [שם_תיק] Relationship Edge List.md` | הטבלה מאפשרת ביקורת טקסטואלית על מקור, סטטוס ורמת ביטחון בלי להסתמך רק על גרף. |

בפועל, העבודה היומית המומלצת היא לפתוח תחילה את `Relationship Map Dashboard.md`, לבחור ממנו את תיק החקירה הרצוי, ורק כאשר נדרש ניתוח רוחבי לפתוח את `Auto Relationship Map.canvas`.

## צבעים וקבוצות לפי תיק

בכל הרצה הסקריפט מקצה צבע עקבי לכל תיק חקירה לפי השדה `related_case`. במפה הגלובלית, הקשתות והקבוצות הוויזואליות מקבלות צבע לפי תיק. אם אותה ישות מופיעה בכמה תיקים, היא עשויה להופיע בצבע רב-תיקי/ניטרלי כדי לא להטעות כאילו היא שייכת רק לתיק אחד.

| רכיב ויזואלי | התנהגות |
| --- | --- |
| קשתות במפה הגלובלית | צבע לפי תיק כאשר `related_case` קיים; קשר ללא תיק נשאר בצביעה לפי confidence/status. |
| קבוצות `Case: ...` | תיבות Group אוטומטיות שמסמנות את אזור הצמתים של כל תיק. |
| `Case Color Legend` | תיבת Legend אוטומטית המסייעת לזהות איזה צבע שייך לאיזה תיק. |
| צמתים קיימים | הצבע הידני שלהם נשמר כדי לא למחוק סידור ויזואלי שכבר בנית. |
| צמתים חדשים | מקבלים צבע ראשוני לפי תיק, או צבע ניטרלי אם הם מופיעים ביותר מתיק אחד. |

> [!warning]
> קבוצות `Case: ...` וה-`Case Color Legend` הן רכיבים אוטומטיים. הן מתרעננות בכל הרצה כדי לשקף את מיקום הצמתים הנוכחי. אם תרצה ליצור קבוצות ידניות משלך, עדיף לתת להן שם שאינו מתחיל ב-`Case:` כדי שהסקריפט לא יזהה אותן כרכיב אוטומטי.

## איך להפעיל דרך Manus

כתוב לי אחת מהפקודות הבאות:

```text
עדכן את מפת הקשרים האוטומטית ב-MANUS CONTROL.
```

או:

```text
סרוק את 04_Relationships וצור Canvas חדש לפי כל הקשרים המאומתים והמשוערים.
```

אני אריץ את הסקריפט, אבדוק אם נוצרו קבצי Canvas תקינים, ואחזיר לך סיכום מפורט: מספר nodes ו-edges במפה הגלובלית ובכל מפת תיק חקירה, צמתים חדשים שנוספו, צמתים קיימים שנשמרו, קשרים חסרים ושדות בעייתיים.

## התנהגות מיזוג ושימור מיקומים

Canvas אינו Dataview. הוא לא מתעדכן בזמן אמת לבד, ולכן צריך להריץ את הסקריפט בכל פעם שרוצים לרענן את מפת הקשרים. עם זאת, הגרסה הנוכחית אינה מאפסת את המפה בכל הרצה.

| רכיב | מקור אמת | מה קורה בהרצה חוזרת |
| --- | --- | --- |
| Relationship Notes | `04_Relationships/*.md` | כל פתק קשר תקין מייצר edge מעודכן. |
| מיקום צמתים | קובץ Canvas ספציפי (גלובלי או תיק חקירה) | צומת קיים עם אותו שם שומר את `x`, `y`, `width`, `height` והצבע הקיים שלו. |
| צמתים חדשים | Relationship Notes חדשים | הסקריפט מוסיף אותם אוטומטית במיקום ראשוני סביב המפה הקיימת. |
| קשתות | Relationship Notes בלבד | הקשתות נבנות מחדש בכל הרצה כדי לשקף את פתקי הקשר העדכניים. |
| צמתים ידניים או קבוצות ידניות | Canvas קיים | צמתים קיימים שאינם חלק מפתקי הקשר נשמרים כדי לא למחוק עבודה ידנית. |
| קבוצות תיק אוטומטיות | הסקריפט | קבוצות בשם `Case: ...` וה-`Case Color Legend` מתרעננות בכל הרצה כדי לשקף את המפה המעודכנת. |

המשמעות המעשית היא שאפשר לעבוד ישירות על קובצי ה-Canvas שנוצרו. אין צורך ליצור עותק ידני רק כדי לשמור את סידור הצמתים. אם תזיז צומת במפה, ואז תריץ שוב את הפקודה `עדכן את מפת הקשרים האוטומטית`, המיקום הידני של הצומת יישמר, וקשרים או צמתים חדשים בלבד יתווספו מעל התשתית הקיימת. אם אתה רוצה תצוגה נקייה ברגע נתון, אל תעבוד מתוך המפה הגלובלית כברירת מחדל; פתח את `Relationship Map Dashboard.md` ובחר ממנו את מפת התיק הרלוונטית.

> [!warning]
> אם משנים ידנית את שם הטקסט של צומת קיים בתוך ה-Canvas כך שהוא כבר לא תואם לשם הישות ב-Relationship Note, הסקריפט יתייחס אליו כאל צומת אחר. כדי לשמר התאמה, שמות הצמתים במפה צריכים להישאר זהים לשמות הישויות בשדות `from_entity` ו-`to_entity`.
